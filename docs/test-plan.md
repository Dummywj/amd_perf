# AMD 长向量性能测试计划

## 测试目标

测试 AMD 服务器在长向量场景下的关键操作性能，重点关注：

- softmax
- fma
- reduce
- gather

测试数据类型：

- FP32
- BF16

核心输出指标：

- `elem/cycle`：平均每个 CPU cycle 处理的元素个数。

## 测试原则

- 先完成 FP32 基准测试，再扩展 BF16。
- 每个操作单独测试，避免不同操作混在同一个计时区间。
- 初始化、内存分配、数据校验不计入性能计时。
- 每个测试包含 warmup，正式结果取多次运行后的稳定值。
- 记录 CPU 型号、编译器、编译参数、线程数、向量长度和 NUMA 绑定方式。

## 建议测试项

| 操作 | 说明 | 数据类型 |
| --- | --- | --- |
| FMA | `out[i] = a[i] * b[i] + c[i]` | FP32、BF16 |
| Reduce | 长向量求和或求最大值 | FP32、BF16 |
| Gather | 按索引从源数组读取元素 | FP32、BF16 |
| Softmax | 长向量 softmax，包含 max、exp、sum、normalize | FP32、BF16 |

## 向量长度

建议覆盖 cache 内和内存带宽主导两类场景：

| 类别 | 建议长度 |
| --- | ---: |
| 小规模 | 1K、4K、16K |
| 中规模 | 64K、256K、1M |
| 大规模 | 4M、16M、64M |

## 结果格式

建议每条结果至少包含：

| 字段 | 说明 |
| --- | --- |
| operation | 操作名称 |
| dtype | 数据类型 |
| vector_length | 向量长度 |
| iterations | 迭代次数 |
| cycles | 计时区间总 cycle 数 |
| elements | 处理元素总数 |
| elem_per_cycle | 平均每 cycle 处理元素数 |
| threads | 线程数 |
| numa | NUMA 绑定方式 |

## 测试顺序

1. 确认编译器和编译参数。
2. 实现 FP32 FMA，验证计时和结果输出链路。
3. 增加 FP32 reduce、gather、softmax。
4. 基于 `avx512_bf16` 增加 BF16 版本。
5. 对每个操作执行多轮测试，观察结果稳定性。
6. 汇总不同操作、数据类型、向量长度下的 `elem/cycle`。

## 注意事项

- 当前机器为双路 AMD EPYC 9684X，测试时需要关注 NUMA 影响。
- 若测试单线程性能，建议固定 CPU core，减少调度抖动。
- 若测试多线程性能，建议分别测试单 NUMA 节点和双 NUMA 节点。
- BF16 测试需要明确输入、输出和中间累加是否使用 BF16 或 FP32。
- softmax 中的 `exp` 实现会显著影响结果，需要在文档中记录使用的是库函数、近似实现还是向量化实现。
